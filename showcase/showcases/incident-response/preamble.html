The Problem

When a terraform apply fails on production, the platform engineer opens HCP Terraform UI, the AWS console, Datadog, and Slack - four tools, three browser tabs. They manually correlate run logs with drift detection results and audit trail entries. Pattern matching that a senior engineer does in 8 minutes takes a junior engineer 25 minutes.

38 minutes average MTTR for P1/P2 run failures. 30 of those minutes are information gathering.

Observe-Correlate-Act

An AI agent ingests HCP Terraform audit log events, run logs, drift detection results, and notification webhooks. Correlates in seconds. Presents ranked root causes with evidence. Suggests remediation runbooks. Engineer approves, agent executes via terraform CLI. Target: under 5 minutes for triage, 50% MTTR reduction.

Product Principles

Three design decisions that shape every stage of this journey.

01
Human Gate on Every Mutation
No terraform apply executes without explicit approval and a mandatory reason field.
02
Evidence Before Conclusion
Every AI root cause links to the raw run logs, state diffs, and audit trail entries that support it. The engineer can drill from summary to source in one click.
03
Every Failure Has a Recovery
Apply failure triggers rollback via state versioning. Verification failure returns to root cause analysis. Approval timeout escalates.

Jobs to Be Done

Three jobs that drive the incident response journey. Each maps to a product principle.

Principle 01

Correlated signals in one view

When an incident fires, I want correlated signals in one view so that I don't spend 20 minutes switching between HCP Terraform UI, AWS console, and Datadog.

Today: the engineer opens four tools across three browser tabs and manually correlates timestamps. Run logs are structured, notification webhooks are not. Cross-referencing takes 8-25 minutes depending on experience.

Principle 02

Verifiable evidence for every root cause

When the AI suggests a root cause, I want to see the raw run logs and state diffs so that I can verify the analysis before approving any remediation.

Trust requires transparency. Each ranked cause links to the specific run logs, drift detection results, and audit trail entries that support it. The engineer validates before acting.

Principle 03

Clear recovery when remediation fails

When a remediation step fails, I want clear recovery options so that I'm not stuck with an unknown system state.

Three recovery paths: retry the failed step, roll back all completed steps in reverse order, or escalate to the next tier with full context preserved. No dead ends.

Recovery Options

When a remediation step fails, the engineer chooses one of three paths. Incident state is preserved across all recovery modes.

Recommended

Retry Step

Single step failed. Fix the issue and retry just that step. Preceding steps are preserved. The runbook resumes from the failure point.

Full Rollback

Execute rollback commands for all completed steps in reverse order. Return to remediation suggestion for a revised plan. System returns to pre-remediation state.

Escalate

Route to Infrastructure Architect for manual intervention. Incident state preserved. Full context attached - correlation data, root cause analysis, and partial execution log.
