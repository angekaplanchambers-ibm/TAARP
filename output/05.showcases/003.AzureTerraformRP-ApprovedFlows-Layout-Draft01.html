Layout Draft 01

Connect First. Operate After.

This first screen frames the journey before any detailed work begins. The operator is already in Azure, already manages infrastructure in HCP Terraform, and needs a clear promise: Azure will discover, validate, and govern without forcing migration or a surprise apply.

Scenario A Single onboarding flow from Azure Portal into connected Terraform stack registration.
Scenario B Operating view after setup: visibility, approvals, policy, support context, and HCP deep links.
Screen Goal Set expectations, reduce migration fear, and move the user into the step-by-step connect flow.

Middle Screen

Each Journey Step Lives In One Tab

This second screen stays structurally stable while the content changes step by step. It keeps the operator in one place, lets you compare flow sequencing quickly, and makes it easy to revise labels or order later.

Scenario A Detail

Connection Journey

Keep the overall frame fixed. The only changing surface is the step content inside the workspace. That gives you one repeatable screen pattern for all onboarding stages.

Simple Starting Rule Use each tab as one stage card, not a fully new page. The layout should be reusable before it gets polished.

Step 1

Enter Terraform In Azure

The operator lands in the Terraform entry point from Azure Portal search and needs immediate confirmation that this flow is meant for existing Terraform users.

User Action

Search for Terraform, open the Azure entry point, and decide whether this is the correct path.

UI Focus

Large welcome panel with clear “existing Terraform” language and a single primary start action.

System Behavior

Reads tenant and role context, then confirms the user can continue onboarding.

User Concern

“Is this a new provisioning workflow, or can I connect what already exists?”

Layout Note

Start with one full-width message area, one explanatory side panel, and a strong continue button.

Risk To Avoid

Do not open with import language. That sends the user down the wrong mental model before trust is established.

Step 2

Understand The Operating Model

This tab is the expectations screen. It explains that HCP Terraform still executes runs and Azure adds discovery, governance, and management.

User Action

Read and confirm how Azure and HCP Terraform divide responsibility.

UI Focus

Three-column value explanation: execution stays in HCP, Azure adds visibility, no automatic apply occurs.

System Behavior

Feature flags and copy configuration shape the rest of the flow.

User Concern

“Will Azure take over Terraform, credentials, or state?”

Layout Note

Use small benefit cards instead of long paragraphs. The screen is about trust framing, not detailed documentation.

Success Signal

The operator should leave this step knowing Azure wraps Terraform rather than replaces it.

Step 3

Connect The HCP Terraform Organization

The operator completes OAuth or SSO, chooses an HCP Terraform organization, and lets Azure validate whether onboarding permissions are present.

User Action

Authenticate, pick an organization, and return to Azure with connected context.

UI Focus

Organization picker, permission status, and a simple readiness indicator.

System Behavior

Checks org membership, permissions, and whether workspaces exist to onboard.

User Concern

“Do I have enough permission to do this once, correctly, without manual backtracking?”

Layout Note

Keep auth, org selection, and permission feedback on one screen so the user does not lose context during redirect.

Risk To Avoid

Do not make permission failure feel like a dead end. Show what role is missing and who can unblock it.

Step 4

Choose The Existing Workspace Path

The operator explicitly selects “Connect existing workspaces.” This locks the rest of the UI to the honest path for already-managed infrastructure.

User Action

Select the existing Terraform path and skip future import capabilities for now.

UI Focus

Two choice cards with strong visual emphasis on the recommended existing-workspace option.

System Behavior

Stores the onboarding mode and filters the remaining experience accordingly.

User Concern

“Will the UI push me into import or migration even though my infra is already managed?”

Layout Note

Use one primary option card and one clearly secondary future-scope card. Avoid equal visual weight.

Success Signal

The user should feel the product is meeting them where they are, not redirecting the problem.

Step 5

Discover Eligible Workspaces

Azure queries HCP Terraform for AzureRM-related workspaces, then presents a short list with enough metadata for confident selection.

User Action

Filter, review, and select the workspaces that should become Azure-connected stacks.

UI Focus

Workspace table with repo, subscription hint, execution mode, and recent run health.

System Behavior

Reads workspace, run, and state metadata to determine likely Azure-managed candidates.

User Concern

“Is Azure showing me the right workspaces, or is this list noisy and incomplete?”

Layout Note

Make this tab feel like a selection workspace: table left, evidence or filters right, action bar pinned low.

Risk To Avoid

Showing too many low-confidence matches will erode trust faster than showing a smaller, clearer list.

Step 6

Map Workspaces To Azure Scope

For each selected workspace, the operator assigns subscription or management group, environment, and owner so Azure can govern it correctly.

User Action

Complete scope and ownership mapping for each workspace in scope.

UI Focus

Structured mapping rows with scope pickers, environment tags, and owner fields.

System Behavior

Prepares managed identity, RBAC, and allowed-scope checks behind the scenes.

User Concern

“If I map this wrong, will Azure attach the workspace to the wrong subscription?”

Layout Note

This tab benefits from row-level validation and a persistent right-side summary of mappings already completed.

Success Signal

Ownership and environment metadata are explicit before the user reaches validation.

Step 7

Validate State And Ownership

This is the trust checkpoint. Azure compares Terraform state references with Azure resource identifiers and confirms the mapped scope without changing infrastructure.

User Action

Review match results, mismatches, and whether the connection is safe to continue.

UI Focus

Non-destructive validation report with “resources found,” “resources mismatched,” and “no changes applied” messaging.

System Behavior

Checks state version data, resource graph evidence, and federated identity readiness.

User Concern

“Is this the step where something hidden changes, or is Azure really just proving what it sees?”

Critical Message

The screen should visibly say that no plan, apply, or state migration is happening here.

Success Signal

The user sees a credible governed-resource count and enough evidence to proceed confidently.

Step 8

Enable Azure Governance Overlays

This is where Azure delivers additive value: policy checks, approval requirements, region or SKU limits, and cost signals layered around Terraform.

User Action

Choose the controls that should apply to these connected stacks now.

UI Focus

Control cards with clear “optional,” “recommended,” and “required for prod” labels.

System Behavior

Saves policy and approval configuration without changing where Terraform plans and applies run.

User Concern

“Are these guardrails useful, or is Azure trying to replace Sentinel and my existing workflows?”

Layout Note

Use grouped policy tiles. This tab should feel like selective layering, not a mandatory configuration gauntlet.

Risk To Avoid

If every control looks mandatory, the flow shifts from additive governance to perceived takeover.

Step 9

Register Connected Stack Resources

Azure writes the connected representation of each Terraform stack so it becomes visible, attributable, and supportable inside Azure.

User Action

Confirm setup and finish registration after validation and governance choices are complete.

UI Focus

Registration summary, per-workspace result list, and explicit confirmation that no Terraform apply ran.

System Behavior

Creates Azure-side inventory records linked to HCP workspace, scope, repo, owner, and governance metadata.

User Concern

“Will I get real Azure visibility after this, or just a confirmation message with no tangible value?”

Layout Note

Show a concrete “before vs after” inventory view here. This is the moment the connected model should become visible.

Success Signal

The user can point to Azure-native objects that now represent the Terraform estate.

Step 10

Operate Connected Stacks

This is the bridge to Scenario B. Azure now shows Terraform-managed infrastructure, policy, run status, approval controls, and support context while HCP Terraform stays the deep action surface.

User Action

Inspect stacks by subscription, review policy and run status, approve or block future applies, and hand issues to support.

UI Focus

Connected stack inventory, status summaries, approval controls, and clear deep links to HCP Terraform.

System Behavior

Reads status and governance metadata across Azure and HCP-linked stack records.

User Concern

“Can Azure add useful operations here without hiding the real Terraform control plane?”

Bridge To Final Screen

This tab is where the tabbed experience transitions into the steady-state single-screen operating view.

Design Constraint

Keep Azure controls visible, but every deep action should still acknowledge HCP Terraform as the authoring and execution source.

Scenario B

The Connected Operating View

The final screen is the steady-state destination. It should feel calmer than the onboarding flow: fewer decisions, more clarity, and obvious separation between Azure governance actions and HCP Terraform deep work.

Azure Does Inventory, policy signals, approvals, support context, ownership metadata, and subscription-level visibility.
HCP Terraform Does Module authoring, plan and apply execution, deeper debugging, and workspace-native Terraform actions.
Primary Outcome Azure gains awareness and governance context without pulling Terraform functionality into Azure.
Iteration Path From here you can add richer tables, approval drawers, and support workflows without changing the overall screen model.

Draft intent: one intro screen, one tabbed journey screen, one steady-state screen. The structure is deliberately simple so the next pass can focus on hierarchy, naming, and which parts deserve richer interaction.
