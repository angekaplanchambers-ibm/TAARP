## Personas

| Persona | Role | Category | Mental Model |
|---------|------|----------|--------------|
| Azure Platform Operator | Connects existing Terraform-managed infrastructure to Azure and manages ongoing governance | User | "Azure should show me what Terraform already manages without forcing me to rebuild or migrate it." |
| Cloud Governance Admin | Defines allowed subscriptions, approvals, and policy guardrails for connected stacks | Admin | "Azure can add governance and visibility, but it must not take over Terraform authoring workflows." |

### Azure Platform Operator

**Category**: User
**Quote**: "My infrastructure is already in Terraform. I need Azure to discover it, validate it, and give me governance without changing how my teams work."

**Context**:
- **Role**: Owns onboarding existing Terraform workspaces into Azure and confirms the connected operating model works after setup.
- **Environment**: Azure Portal, HCP Terraform, Azure subscriptions and management groups, Git-backed Terraform workspaces.
- **Frequency**: Initial onboarding happens per organization or environment; operating tasks happen weekly or during governance reviews and run approvals.

**Goals**:
- Connect existing Terraform workspaces from Azure Portal
- Map those workspaces to the right Azure scopes and owners
- Verify that Azure governance wraps Terraform without forcing state migration or a fresh apply
- Use Azure for visibility, approvals, policy, and support context after setup

**Pain Points**:
- Existing Terraform users do not want import promises that only work for greenfield cases
- Credentials, identity mapping, and subscription alignment are easy places to lose trust
- Post-setup value must be obvious or the connection flow feels like paperwork

**Design Implications**:
- The onboarding path must clearly separate existing-workspace connection from unmanaged-resource import
- Validation steps must prove trust without making infrastructure changes
- The handoff from setup to steady-state operations must be explicit in the UI

### Cloud Governance Admin

**Category**: Admin
**Quote**: "I need Azure controls around Terraform runs, but I do not want to replace HCP Terraform as the execution and authoring system."

**Context**:
- **Role**: Defines approved scopes, policies, and approval flows for connected Terraform stacks.
- **Environment**: Azure Policy, RBAC, subscriptions, management groups, support and audit workflows.
- **Frequency**: Engaged during onboarding design, production rollout, and ongoing exceptions.

**Goals**:
- Ensure connected workspaces only target approved Azure scopes
- Apply governance controls before high-risk applies
- Give support and audit teams enough metadata to troubleshoot connected stacks

**Pain Points**:
- Governance overlays often get interpreted as migration requirements
- Subscription and ownership mapping can become inconsistent across teams
- Support workflows break if Azure has only partial context about Terraform-managed resources

**Design Implications**:
- Governance choices should be additive and clearly optional where appropriate
- Azure-side resources must preserve links back to the HCP Terraform source of truth
- Audit and support metadata need to be created at registration time, not reconstructed later

## Jobs To Be Done

When I already manage Azure infrastructure through HCP Terraform and want Azure-native visibility and governance,
I want to connect my existing Terraform workspaces in Azure Portal and register them as Azure-aware resources,
So that I can govern, inspect, approve, and support those stacks from Azure without changing how teams author or run Terraform.

- **Functional**: Discover eligible Terraform workspaces, map them to Azure scopes, validate state ownership, register connected stack resources, and enable ongoing governance actions.
- **Emotional**: Confidence that Azure is validating and wrapping the existing Terraform estate rather than mutating it unexpectedly.
- **Social**: Show platform, governance, and support teams a shared operating model with clear ownership and audit context.

## Critical User Journeys

**Supporting 1**: As a Cloud Governance Admin, I want to apply Azure-side controls to connected Terraform stacks so that policy, approvals, and support workflows work without replacing HCP Terraform.

## Per-Stage Detail

### Stage 1: Enter Terraform in Azure
**User Action**: The operator searches Azure Portal for Terraform, filters to the relevant resource experience, and opens the Terraform entry point.
**Agent Action**: Azure presents the Terraform landing page and confirms this experience is for Azure-governed Terraform usage.
**Touchpoint**: Azure Portal search results and Terraform landing page.
**System Calls**: Azure Portal shell queries resource and service navigation metadata for Terraform-related entries.
**Data In**: Tenant context, signed-in user, portal navigation state.
**Data Out**: Terraform landing context with selected tenant and subscription context.
**Validation**: Confirm the user has enough Azure access to continue onboarding.
**Failure Mode**: Terraform entry point is unavailable for the tenant or cloud environment. Recovery: show prerequisites and exit path.
**Duration**: Under 1 minute.

### Stage 2: Understand the model
**User Action**: The operator reads the intro view that explains HCP Terraform remains the execution plane while Azure adds discovery, governance, and management.
**Agent Action**: Azure sets expectations: no Terraform CLI install is required for onboarding, no migration is implied, and no apply will run automatically.
**Touchpoint**: Splash intro screen.
**System Calls**: None beyond portal content and feature-flag checks.
**Data In**: Product messaging content, feature availability.
**Data Out**: User acknowledgement to continue.
**Validation**: Require explicit acknowledgement before moving into connection.
**Failure Mode**: The user does not understand whether existing workspaces are supported. Recovery: expose examples and path labels for existing infra vs future import.
**Duration**: 1-2 minutes.

### Stage 3: Connect HCP Terraform org
**User Action**: The operator selects to connect HCP Terraform, completes OAuth or SSO, and chooses an HCP Terraform organization.
**Agent Action**: Azure starts the HCP Terraform authorization flow, receives an access token, lists organizations the user can onboard, and checks for onboarding permissions.
**Touchpoint**: OAuth redirect, HCP sign-in, organization picker.
**System Calls**:
- `GET /api/v2/account/details`
- `GET /api/v2/organizations/{organization_name}`
- `GET /api/v2/organizations/{organization_name}/entitlement-set`
**Data In**: HCP identity, selected organization.
**Data Out**: Authorized HCP organization context and permission result.
**Validation**: Confirm the user is an org admin or holds onboarding permissions, and that the organization contains workspaces.
**Failure Mode**: OAuth fails or the user lacks permissions. Recovery: return to org picker with admin requirement guidance.
**Duration**: 1-3 minutes.

### Stage 4: Choose onboarding mode
**User Action**: The operator chooses the existing-infrastructure path and explicitly selects "Connect existing workspaces."
**Agent Action**: Azure suppresses unmanaged-resource import flows and narrows the rest of the experience to existing Terraform workspaces only.
**Touchpoint**: Onboarding mode selector.
**System Calls**: Internal onboarding state service persists the selected mode.
**Data In**: Mode selection.
**Data Out**: Onboarding context set to existing-workspace connection.
**Validation**: Require a mode choice before discovery starts.
**Failure Mode**: The user selects the wrong mode. Recovery: allow switching modes before registration.
**Duration**: Under 1 minute.

### Stage 5: Discover eligible workspaces
**User Action**: The operator reviews discovered HCP Terraform workspaces, filters to Azure-related ones, and selects the workspaces to connect.
**Agent Action**: Azure queries HCP Terraform for workspaces, provider usage, recent run health, and visible subscription hints, then presents an operator-friendly list.
**Touchpoint**: Workspace discovery table.
**System Calls**:
- `GET /api/v2/organizations/{organization_name}/workspaces`
- `GET /api/v2/workspaces/{workspace_id}`
- `GET /api/v2/workspaces/{workspace_id}/runs`
- `GET /api/v2/workspaces/{workspace_id}/current-state-version`
**Data In**: Authorized org context, workspace metadata, run metadata, state metadata.
**Data Out**: Candidate workspace list with name, repo, provider signals, subscription hints, and last apply status.
**Validation**: Show only workspaces with enough evidence that they target AzureRM-managed infrastructure, and flag unsupported execution modes if needed.
**Failure Mode**: Discovery returns too many ambiguous workspaces or missing subscription signals. Recovery: let the operator filter manually and inspect evidence.
**Duration**: 10-60 seconds depending on workspace count.

### Stage 6: Map workspace to Azure scope
**User Action**: For each selected workspace, the operator maps it to a subscription or management group, environment, and owner.
**Agent Action**: Azure loads available subscriptions and management groups, checks allowed scopes, and prepares Azure identity and RBAC prerequisites.
**Touchpoint**: Scope mapping and ownership form.
**System Calls**:
- `GET /subscriptions?api-version=2020-01-01`
- `GET /providers/Microsoft.Management/managementGroups?api-version=2020-05-01`
- `PUT /subscriptions/{subscriptionId}/resourceGroups/{resourceGroupName}/providers/Microsoft.ManagedIdentity/userAssignedIdentities/{identityName}?api-version=2023-01-31`
- `PUT /subscriptions/{subscriptionId}/providers/Microsoft.Authorization/roleAssignments/{roleAssignmentId}?api-version=2022-04-01`
**Data In**: Selected workspaces, Azure scope selections, environment and owner metadata.
**Data Out**: Proposed workspace-to-scope mapping plus managed identity and RBAC plan.
**Validation**: Confirm selected subscriptions or management groups are allowed and ownership metadata is complete.
**Failure Mode**: Scope is not allowed or RBAC is insufficient. Recovery: block the mapping and show the required Azure role or allowed-scope policy.
**Duration**: 2-5 minutes for several workspaces.

### Stage 7: Validate state and ownership
**User Action**: The operator reviews a non-destructive validation summary and decides whether the discovered infrastructure matches expectations.
**Agent Action**: Azure inspects workspace state references, compares Azure resource identifiers and subscription ownership, and verifies the mapped credentials can work with federated auth.
**Touchpoint**: Validation summary screen.
**System Calls**:
- `GET /api/v2/workspaces/{workspace_id}/current-state-version`
- `GET /api/v2/state-versions/{state_version_id}`
- `POST /providers/Microsoft.ResourceGraph/resources?api-version=2021-03-01`
- `PUT /subscriptions/{subscriptionId}/resourceGroups/{resourceGroupName}/providers/Microsoft.ManagedIdentity/userAssignedIdentities/{identityName}/federatedIdentityCredentials/{credentialName}?api-version=2023-01-31`
**Data In**: State metadata, Azure resource IDs, selected mappings, identity configuration.
**Data Out**: Validation report showing matched resources, mismatches, and whether onboarding can continue.
**Validation**: Confirm state exists, targeted Azure resources are visible, and resource IDs match the intended subscription or management group.
**Failure Mode**: State is stale, resource IDs do not match, or federated identity setup fails. Recovery: pause onboarding and return a mismatch report without applying changes.
**Duration**: 30-120 seconds.

### Stage 8: Enable governance overlays
**User Action**: The operator, with governance input as needed, chooses Azure-side controls such as policy evaluation, approval requirements, region limits, or cost signals.
**Agent Action**: Azure records governance preferences and translates them into Azure policy or approval configuration while preserving HCP Terraform as the run engine.
**Touchpoint**: Governance configuration screen.
**System Calls**:
- `PUT /subscriptions/{subscriptionId}/providers/Microsoft.Authorization/policyAssignments/{policyAssignmentName}?api-version=2023-04-01`
- Azure approval workflow configuration through internal governance services
**Data In**: Selected governance controls, mapped scopes, environment criticality.
**Data Out**: Governance configuration plan linked to each connected workspace.
**Validation**: Make clear which controls are optional, recommended, or required for production scopes.
**Failure Mode**: Governance rules are too restrictive and would block standard workflows. Recovery: preview the impact before saving and allow staged rollout.
**Duration**: 2-10 minutes depending on policy complexity.

### Stage 9: Register connected stack resources
**User Action**: The operator confirms setup and finishes registration.
**Agent Action**: Azure creates Azure-side records for each connected Terraform stack, links them to workspace, scope, repo, owner, and governance metadata, and exposes them as first-class Azure-managed entries.
**Touchpoint**: Registration confirmation view.
**System Calls**:
- Internal RP registration service for connected stack records
- Azure ARM persistence for stack metadata records and inventory indexing
**Data In**: Approved mappings, validation report, governance configuration, HCP workspace metadata.
**Data Out**: Connected stack resources with Azure and HCP linkage metadata.
**Validation**: Confirm registration created Azure-visible records without triggering Terraform plan or apply.
**Failure Mode**: Registration succeeds partially and leaves some workspaces unlinked. Recovery: show per-workspace registration results and allow safe retry.
**Duration**: 30-90 seconds.

### Stage 10: Operate connected stacks
**User Action**: After setup, the operator and governance admin use Azure to inspect Terraform-managed infrastructure by subscription, review policy and run status, approve or block future applies, and route issues to support with context. Deep Terraform authoring and debugging still happen in HCP Terraform.
**Agent Action**: Azure surfaces connected stack inventory, status, policy, ownership, and support metadata, and links back to HCP Terraform for deep actions.
**Touchpoint**: Azure connected-stacks inventory, detail pages, approvals, and support entry points.
**System Calls**:
- HCP Terraform workspace and run status reads for linked stacks
- Azure support and audit services for incident and case context
- Azure governance services for policy and approval decisions
**Data In**: Registered stack resources, run status, policy results, support metadata.
**Data Out**: Ongoing operational view of connected Terraform stacks inside Azure.
**Validation**: Ensure ongoing Azure actions remain additive and do not sever HCP Terraform as the source of execution and authoring.
**Failure Mode**: Users assume Azure replaced HCP Terraform. Recovery: preserve deep links, action labels, and role boundaries in every connected-stack view.
**Duration**: Ongoing operational use.
