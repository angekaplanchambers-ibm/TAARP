# User Journey

## Scenario A: I’m in Azure Portal, I select TF as an IaC Service, and already have Infra managed by TF. How do I connect?
### Step 1: Initial Setup in Azure Portal (setting expectations)
Search > Terraform, Filter by Resources tag.
Select > Terraform.
Click > Get Started and see a splash intro view.

Azure explains:
- TF runs in Hashi-managed control plane (HCP)
- Azure provides discovery, governance and management
- No TF CLI is required to install.

### Step 2: Connect to HCP TF (establishing trust)
User choice (either or):
- (Optional) Create a new one
- Connect an existing HCP Terraform organization

If connecting:
- Azure initiates OAuth / SSO
- User selects the HCP org

Azure verifies:
- The user is an org admin or has onboarding permissions
- The org contains Terraform workspaces

### Step 3: Choose Onboarding Mode for Existing Infra (This keeps the experience honest and avoids overpromising import magic)
Azure presents explicit options for existing Terraform users:
- Option A (SELECTED): Connect existing workspaces (most common) - “My infrastructure is already managed by Terraform.”
- Option B: Import unmanaged Azure resources - (Future / limited scope, clearly marked)

### Step 4: Discover Existing Terraform Workspaces
Azure (via RP) queries HCP Terraform for:
- Workspaces using the Azure RM provider
- Target Azure subscriptions
- Execution mode (remote/local)
- Current run health
Azure shows a workspace list:
- Workspace name
- Repo (if connected)
- Subscription detected
- Last apply status
User selects which workspaces to connect to Azure.

### Step 5: Map Workspaces to Azure Scope
For each selected workspace, the user maps:
- Azure subscription or management group
- Environment (dev / test / prod)
- Owner (team or service)
Azure validates:
- The workspace’s Azure credentials
- That the target subscription is allowed
Behind the scenes:
- Azure creates or assigns a managed identity
- Azure configures OIDC trust with HCP Terraform
- Terraform workspace is updated to use Azure federated auth
Service principals and secrets are eliminated.

### Step 6: Verify State & Resource Ownership (This step is about trust, not change)
Azure performs a non-destructive validation:
- Confirms Terraform state exists and is healthy
- Detects Azure resources in state
- Verifies Azure subscription/resource IDs match
User sees a summary:
- “X resources will be governed by Azure Terraform RP”
- No changes applied
- No state migration required

### Step 7: Enable Azure Governance (Optional? Recommended?)
First point at which a user experiences new/tangible value from this interaction, that would not be experienced singularly in Terraform.
User can now opt into Azure-side controls:
- Azure Policy evaluation before runs
- Approval requirements (e.g., prod applies)
- Region / SKU limits
- Cost signals
Important to Note:
- Nothing forces a reapply
- Terraform policies (Sentinel) remain intact
- Azure governance wraps, not replaces, Terraform

### Step 8: Register Terraform Workspaces as Azure Resources (This is the “connected” moment)
Azure now creates Azure RP resources representing the Terraform stacks.
Examples:
- Microsoft.Terraform/stacks
- Microsoft.Terraform/environments
Each one links to:
- HCP Terraform workspace
- Azure subscription(s)
- Repo metadata
From this point forward:
- Terraform managed resources are attributed in Azure
- Azure gains native awareness, attribution and governance context for TF-managed Infra, enabling support, auditing and visibility. Doing all of this without moving the actual Terraform functionality into Azure.

### Step 9: Confirm & Finish
Confirmation screen:
Existing Terraform infrastructure is now connected to Azure
Azure now shows:
- Terraform stacks in the Portal
- Last run status
- Ownership metadata
- Links to HCP for deep actions
No Terraform apply has been triggered unless the user explicitly asks.

## Scenario B: I’m in Azure Portal and have confirmation of the setup. What can I do now? (Additive, not migratory)
From Azure:
- View all Terraform-managed infra by subscription
- Approve or block future applies
- See policy and run status
- Hand issues to Microsoft support with context
From HCP Terraform:
- Continue authoring modules and configs
- Run plans/applies
- Debug deeply
No workflows are broken.
