## TL;DR
HashiCorp is exploring a Terraform Resource Provider (RP) on Azure to move Terraform from a third-party tool position to a first-class Azure service experience. The current preferred direction is a hybrid model where Azure is the system of intent, governance, support, and discovery, while HCP Terraform remains the execution control plane. Key unresolved decisions include hosting model trade-offs, billing design, compliance constraints, and implementation scope/effort.

## Context
Consolidated synthesis of four source notes discussing strategic motivation, technical options, partner model comparisons, and onboarding journey design for Terraform RP on Azure.

## Key Discussion Points

### Strategic Intent and Business Drivers
- The core goal is first-class Terraform presence in Azure Portal and service taxonomy to reduce day-0 friction and increase adoption.
- Expected GTM impact is stronger Microsoft field alignment under hybrid 1P/resell patterns.
- The model is positioned as a conversion path from community/clickops usage to commercial Terraform.
- AI-focused Azure workflows (Copilot/AI Foundry) are considered a growth vector that benefits from RP integration.
- Joint support is a major driver: Microsoft first-line triage with structured escalation to HashiCorp.

### Decision Framing and Delivery Models
- Notes compare RPaaS, Hybrid 1P/Resell, and Full 1P, with trade-offs in IP ownership, compliance model, billing, support agreements, and timeline.
- Technical option comparison is centered on:
- Option A: Self-hosted TFE in Azure tenant.
- Option B: RP shim to HCP Terraform SaaS.
- Option A favors in-tenant UX and compliance control but increases implementation and operations burden.
- Option B favors speed and lower build/operations cost but introduces portal-to-HCP UX discontinuity and potential compliance objections.

### Technical Workstreams
- RP API design and TypeSpec authoring for Azure RP contract compliance.
- RP registration and manifest rollout.
- Declarative portal extension for onboarding and management UX.
- Mandatory Azure service integration points: AAD, Azure Monitor, and billing surfaces.
- Potential private SaaS offer and marketplace test/review workflow.

### User Journey and Product Behavior
- Journey design stresses that Terraform is not moved into Azure; Azure wraps governance and visibility around existing Terraform workflows.
- Existing-workspace onboarding is explicit and non-destructive:
- Connect HCP org via OAuth/SSO.
- Discover eligible workspaces.
- Map workspace-to-subscription/environment/owner.
- Configure managed identity + OIDC federation.
- Validate state/resource alignment without forced apply.
- Optional governance controls are additive (policy checks, approvals, limits, cost signals).
- Connected state is represented as Azure RP resources linked to HCP workspaces.

===== Motivation for building a Terraform RP on Azure.docx =====
HashiCorp's motivation for building a Terraform RP on Azure
Today, Terraform on Azure behaves like a third-party tool, operating adjacent to the control plane - and this structural positioning limits GTM alignment and seamless customer adoption. The RP model being considered is a structural shift: Terraform becomes a first-class citizen in Azure's service taxonomy.
Intent of this document -
        •       Ensure internally, the HashiCorp team is aligned on why we are doing this.
        •       Help with any internal prioritization
        •       Share a redacted version of this document to help Microsoft team understand our drivers.

Terraform as an Azure Resource Provider (RP)
Following strategic discussions with Microsoft leadership, we are kicking off investigations to build a Terraform Resource Provider (RP) on Azure. This approach, recommended and supported in principle by Azure leadership, allows Terraform to function as a native-feeling service within the Azure Portal, similar to Oracle Database@Azure or Anyscale.
The primary motivation is to secure a first-class status for Terraform on Azure, creating a seamless top-of-funnel experience that drives user adoption and accelerates the transition from community usage to paid Enterprise IaC Tooling.
Additionally, data suggests that transitioning customers to commercial tooling appreciably increases their Azure resource consumption as it easily allows customers to scale, creating a shared incentive for Microsoft.
Strategic drivers for HashiCorp
1. Establishing a Native Day-0 Experience (Top-of-Funnel)
The current friction for new Azure users, often in TF-CE, involves installing CLIs and managing state files manually. By building a native RP, HashiCorp can embed commercial Terraform directly into the Azure Service Catalog and Portal UI.
        •       The Click-to-Deploy Journey: Users can select a solution (e.g., a secure VM cluster) in the Azure Portal and deploy it using a Terraform resource type behind the scenes.
        •       Reduced Friction: This creates a seamless Day 0 experience where the complexity of the binary is abstracted, similar to how Bicep operates over ARM.
        •       Goal: Introduce thousands of developers to Terraform implicitly, making it the de facto standard for IaC on Azure.
2. Getting alignment with MS GTM engine to drive sales
With the hybrid 1P approach, we will unlock the MS field team to sell Terraform. There will be GTM alignment towards the goal of selling Terraform.
        •       We have struggled in the past to get Microsoft to allocate any GTM resources let alone selling HashiCorp products. With the Hybrid 1P path, Microsoft sales team will be responsible for selling Terraform.
        •       None of the control planes of our services are hosted on Azure. In the past, Microsoft has pushed for these in every conversation so much so that for all marketplace listings on Azure we have had to go through a special exception process each year. This should elevate those discussions moving forward.
        •       Given Microsoft has 2 of their own proprietary IaC tools (ARM and Bicep) that are offered free to their customers, they will not be cannibalizing those by selling Terraform over those. However, for customers who are using or have expressed strong affinity to Terraform, MS sales could offset their own quota by selling Terraform.
        •       This approach would elevate TF to a first-class citizen (same as ARM and Bicep) in Azure allowing us to collaborate with non-technical teams on ancillary areas such as documentation, training, tutorials, etc.
3. Smooth transition from Community to Enterprise
The RP serves as a nudge to migrate existing ClickOps users and Terraform Community users toward the commercial version.
        •       The Upsell Path: Once a customer reaches scaling limits or requires governance, the RP integration offers a direct, automated migration path to Enterprise IaC Tooling (HCP Terraform).
        •       For Platform Engineers, the RP transforms unmanaged resources into a governed state, creating immediate value that justifies commercial spend.
4. Capitalizing on the AI-First Infrastructure Wave
Microsoft's current strategic priority is AI (Copilot, AI Foundry). An Azure RP allows Terraform to integrate natively with these high-growth areas.
        •       AI-Led Deployment: Users can prompt Azure Copilot to generate infrastructure; the RP enables the seamless execution of that code, supported by a Terraform MCP server for best practices.
        •       AI Foundry Integration: The RP can facilitate the rapid, secure provisioning of complex Azure AI Foundry environments, removing infrastructure bottlenecks for AI developers.
5. Proper support integration between the 2 companies
Traditionally the hand off of customer issues between Microsoft and HashiCorp support has never happened including some attempts to formalize between the 2 support teams around 2 years back. The existing process is very convoluted and manual leading to significant avoidable delays for joint customers.
        •       Joint Support: In this hybrid 1P model, Microsoft will become the primary, and will triage to HashiCorp. In this way there will be a reasonable hand off for the customers and will lead to a faster turn around on addressing joint customer issues.
        •       One additional benefit will be that there will be fewer instances of Microsoft customers not being HashiCorp commercial customers as those originating from RP will be all HashiCorp commercial customers.
        •       Warm hand off may still not be possible as it may require snapping to Microsoft's support process and tooling including in worst cases getting physical hardware to view customer cases.
Technical and Business Viability
We have kicked off commercial and technical streams to get a better understanding of the work needed here but we do not have any estimates to share yet. Doing this investigation also impacts some of the current work planned for CY26.
        •       Feasibility: Microsoft has confirmed technically this is achievable and similar to paths taken by Oracle and Anyscale. While Microsoft Core Engineering will not build it, they will provide guidance on the Private RP platform.
        •       Implementation: The RP will function as an orchestration layer, serving as a control-plane frontend that allows Terraform workflows to be initiated and monitored from within Azure. The RP would forward execution requests to HCP Terraform, which performs planning, state management, and provider execution. However, the extent of engineering work needed on HashiCorp side is still unknown until additional estimation is done by Terraform PM and Engineering teams.
        •       Business Model: Unlike a full 1P model, this allows HashiCorp to retain control over IP and security standards while leveraging the Azure Portal for distribution.
Conclusion
Building a Terraform RP on Azure is the most viable path to deeper integration with Microsoft/Azure without actual hosting on Azure. It secures HashiCorp's goal of creating a high-volume funnel for Enterprise IaC adoption while creating customer value.

Technical investigations doc
Intent of this doc -
The goal is to use this doc to capture the technical choices, pros and cons, debt needed to achieve a Terraform resource provider on Azure. This is not a design doc. However we will explore the design choices that will help us do a more accurate assessment of the technical work needed.
Target User Personas for a Terraform Azure Resource Provider
The target users for utilizing Terraform as a resource provider on Azure generally fall into roles heavily involved in cloud infrastructure automation, deployment, and management.
Assumption -
        •       Current Terraform customers and heavy users are unlikely to search for and use this RP on Azure.
        •       Azure only users with no need to manage resources outside of Azure will also be unlikely candidates.
1. Cloud Infrastructure Engineer/DevOps Engineer
        •       Goal: Automate provisioning, configuration, and management of Azure infrastructure in a repeatable, efficient, and consistent manner.
        •       Key needs: IaC practices, state management, modules, CI/CD integration.
2. Platform Engineer
        •       Goal: Create reusable modules and templates with governance guardrails for self-service.
        •       Key needs: governance, compliance, module lifecycle, shared services.
3. Developer (you build it, you run it)
        •       Goal: Quickly stand up and tear down environments with minimal infrastructure overhead.

Decision tree
Option A: Terraform Enterprise on the Azure Tenant (Self-Hosted)
Pros:
- Better in-Azure control-plane and web UX.
- Greater control/compliance posture for regulated environments.
- Azure billing clarity for infrastructure consumed by deployment.
Cons:
- Larger engineering scope for provisioning/config/integration.
- Ongoing operations and maintenance burden.
- Customer/account onboarding complexity.

Option B: Shim to HCP Terraform (SaaS)
Pros:
- Lower engineering effort.
- HashiCorp-managed maintenance and upgrades.
- Faster time to market.
Cons:
- UX continuity gap from Azure portal into HCP UI.
- Reduced customer control versus self-hosted.
- Connectivity/compliance objections for some enterprises.
- Potentially less deep Azure-native integration.

Technical implementation areas
- Develop RP API and TypeSpec to satisfy common RP contract.
- Register RP and resource types.
- Roll out RP manifest.
- Build portal extension and setup wizard.
- Integrate AAD for RBAC.
- Integrate Azure Monitor.
- Define billing model.

Questions for Microsoft
- Does an RP have to be SaaS-only?
- If PaaS is allowed, how much control plane must run in Azure?

===== User Journey.docx =====
Core positioning:
We are not moving Terraform into Azure. We are attaching Azure governance, visibility, and support to existing Terraform workflows.

Value proposition:
Users return to Azure because that is where intent, governance, visibility, and support live. HCP remains the execution system.

Scenario A: Existing Terraform user connects from Azure portal
1) Initial setup
- User discovers Terraform in Azure portal and starts onboarding.
- Azure sets expectation that execution remains in HashiCorp control plane.
2) Connect to HCP Terraform org
- OAuth/SSO flow and org authorization checks.
3) Choose onboarding mode
- Preferred mode is connect existing workspaces.
- Import unmanaged resources is future/limited scope.
4) Discover workspaces
- Query for AzureRM-based workspaces, target subscriptions, run health.
5) Map to Azure scope
- Map subscription/management group, environment, owner.
- Configure managed identity and OIDC federation.
6) Verify state and ownership
- Non-destructive validation only.
- No forced migration or apply.
7) Enable Azure governance
- Optional controls: policy checks, approvals, limits, cost signals.
8) Register as Azure RP resources
- Represent workspaces/stacks as Azure resources linked to HCP metadata.
9) Confirm and finish
- Show run status, ownership metadata, support context, links to HCP deep actions.

Scenario B: Post-setup day-2 use
From Azure:
- View Terraform-managed infra by subscription.
- Approve/block future applies.
- Review policy/run status.
- Engage Microsoft support with context.
From HCP:
- Continue authoring modules/config.
- Execute plans/applies.
- Perform deep debugging.
No existing workflows are broken.
