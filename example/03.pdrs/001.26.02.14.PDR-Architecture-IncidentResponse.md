# PDR-001: Incident Response Dashboard - Architecture

**Status**: Active
**Date**: 2026-02-18
**Author**: Maya Chen
**Depends on**: [002.IncidentResponse-JourneyMap](../02.strategy/002.26.02.14.IncidentResponse-JourneyMap.md)

## Problem Statement

When a terraform apply fails or drift is detected across workspaces on HCP Terraform,
Platform Engineers want AI-correlated signals from run logs, audit events, and drift detection results to triage root cause and execute approved remediation from a single dashboard,
So that mean time to resolution stays under SLA without manual cross-referencing of run output across the HCP Terraform UI, AWS Console, and Datadog dashboards.

Today: engineers scan raw run logs across 2-4 browser tabs, manually correlate timestamps and resource addresses, escalate via Slack with incomplete context, and approve remediations verbally. Correlation alone takes 10-15 minutes. The target is under 60 seconds.

---

## Design Pattern

**Observe-Correlate-Act with Human Gate**

The system follows a three-phase loop with an explicit human approval gate before any mutating action.

```
OBSERVE           CORRELATE              ACT
┌─────────┐      ┌──────────────┐      ┌──────────────┐
│ Ingest   │─────▶│ Correlate &  │─────▶│ Suggest      │
│ Signals  │      │ Rank Causes  │      │ Remediation  │
└─────────┘      └──────────────┘      └──────┬───────┘
                                              │
                                     ┌────────▼────────┐
                                     │  HUMAN GATE     │
                                     │  (Approval +    │
                                     │   Authorization)│
                                     └────────┬────────┘
                                              │
                                     ┌────────▼────────┐
                                     │  Execute &      │
                                     │  Verify         │
                                     └─────────────────┘
```

**Why this pattern**: Production infrastructure changes are high-consequence. A wrong force-unlock or state manipulation can cascade to downstream workspace failures and broken dependencies across the infrastructure graph. The agent adds value in the Observe and Correlate phases (speed, pattern matching across high-volume run and audit data). The human gate ensures accountability for mutating actions.

The pattern decomposes into the 10 journey stages:

- **Observe**: Stages 1-3 (Alert Trigger, Signal Correlation, Triage View)
- **Correlate**: Stage 4 (Root Cause Analysis)
- **Act (suggest)**: Stage 5 (Remediation Suggestion)
- **Human Gate**: Stage 6 (Approval & Authorization)
- **Act (execute)**: Stages 7-8 (Playbook Execution, Execution Monitoring)
- **Verify**: Stages 9-10 (Verification, Incident Closure)

---

## Journey Flow

The 10 stages map to the journey map in [002.IncidentResponse-JourneyMap](../02.strategy/002.26.02.14.IncidentResponse-JourneyMap.md). This section defines the architectural role of each stage.

| Stage | Architectural Role | Primary Actor | Data Transformation |
|-------|-------------------|---------------|---------------------|
| 1. Alert Trigger | Ingestion | Agent | Raw signals → AlertEvent |
| 2. Signal Correlation | Correlation | Agent | AlertEvent[] → IncidentEnvelope |
| 3. Triage View | Presentation | Engineer | IncidentEnvelope → TriageSummary (rendered) |
| 4. Root Cause Analysis | Analysis | Agent + Infra Architect | IncidentEnvelope + raw data → RootCauseAnalysis |
| 5. Remediation Suggestion | Planning | Agent | RootCauseAnalysis → RemediationPlan |
| 6. Approval & Authorization | Authorization | IC + Engineer | RemediationPlan → ApprovalRecord |
| 7. Playbook Execution | Mutation | Agent (supervised) | ApprovalRecord + RemediationPlan → ExecutionLog |
| 8. Execution Monitoring | Observation | Engineer + IC | ExecutionLog (streaming) → MonitoringState |
| 9. Verification | Validation | Agent + Engineer | ExecutionLog + system state → VerificationResult |
| 10. Incident Closure | Record-keeping | Engineer + IC | All prior outputs → IncidentRecord |

**Key constraint**: No stage in the Act phase (7-8) executes without a completed ApprovalRecord from Stage 6. This is enforced at the API level, not just the UI.

## Editability Model

Engineers can edit at defined points. Edits propagate forward but do not retroactively change upstream artifacts.

### What engineers can edit

| Editable Element | When | How Changes Propagate |
|-----------------|------|----------------------|
| Alert thresholds (severity mapping) | Before or during incident | Agent re-evaluates incoming alerts against new thresholds. Active incident re-triaged if thresholds change. |
| Signal correlation grouping | Stage 2 (review) | Engineer can ungroup falsely correlated alerts or manually group missed correlations. Agent re-runs root cause analysis on modified grouping. |
| Remediation playbook parameters | Stage 5 (review) | Changed parameters flow to execution. Agent validates parameter types and ranges before accepting edits. |
| Playbook step order | Stage 5 (review) | Agent re-validates dependency order. Rejects reorderings that break step dependencies (e.g., cannot apply plan before generating it). |
| Add/remove playbook steps | Stage 5 (review) | Added steps require command, expected output pattern, and rollback command. Removed steps must not break dependencies. |
| Approval reason | Stage 6 | Written to audit trail. No downstream propagation beyond the record. |
| Verification check overrides | Stage 9 | Engineer can mark a failed auto-check as "accepted with justification." Justification is required (min 20 chars) and written to audit trail. |

### What engineers cannot edit

- Agent correlation logic (they can override results, not the algorithm)
- Confidence scores (read-only; agent-generated)
- Audit trail entries (append-only)
- Executed playbook steps (immutable after execution; new steps can be added)
- HCP Terraform Teams role assignments (managed by org settings, not the dashboard)

## Agent Responsibilities vs. User Decisions

| Responsibility | Owner | Rationale |
|---------------|-------|-----------|
| Ingest and parse run logs, audit events, and drift results | Agent | High-volume, continuous. Human cannot keep pace with streaming run output across dozens of workspaces. |
| Correlate alerts into incident envelopes | Agent | Pattern matching across multiple data sources in time windows. Agent does this in seconds; engineer takes 10-15 minutes. |
| Rank root causes with confidence scores | Agent | Agent applies statistical correlation. Engineer verifies against domain knowledge. |
| Select remediation playbook | Agent | Agent matches root cause pattern to playbook library. Engineer reviews selection. |
| Populate playbook parameters | Agent | Agent fills from incident context. Engineer verifies and edits. |
| Flag high-risk steps for review | Agent | Agent applies risk classification rules. Infrastructure Architect reviews flagged steps. |
| Execute playbook steps | Agent (supervised) | Agent runs commands. Engineer watches output. Agent halts on unexpected output. |
| Run automated verification checks | Agent | Agent executes health checks. Engineer confirms results. |
| **Decide** severity classification override | Engineer | Engineer can override agent's severity assessment based on business context the agent lacks. |
| **Decide** to escalate to Infrastructure Architect | Engineer | Engineer judges when root cause analysis exceeds their expertise. |
| **Decide** to approve remediation | Incident Commander | Approval is a human accountability decision. Agent cannot self-approve. |
| **Decide** to override a failed verification check | Engineer | Requires justification. Agent cannot override its own checks. |
| **Decide** to initiate rollback | Engineer | Rollback is a high-consequence decision. Agent suggests rollback when execution fails; engineer decides. |
| **Decide** to close the incident | Engineer + IC | Closure is a human judgment. Agent generates the summary; humans sign off. |

## Rollback Model

Every mutating operation in the Act phase has a defined rollback path.

### Rollback by stage

| Stage | Mutating Action | Rollback Action | Reversibility |
|-------|----------------|-----------------|---------------|
| 7. Playbook Execution | `terraform state push <prior>` (state restoration) | Re-run `terraform state pull` and diff against known-good state. If divergent, push the snapshot taken before the operation. | Full - state is versioned and prior versions are retained in HCP Terraform. |
| 7. Playbook Execution | `terraform apply -destroy -target=<resource>` (targeted destroy) | Re-run `terraform apply` to recreate the resource from configuration. Resource data (e.g., instance metadata, attached volumes) may be lost. | Partial - infrastructure is recreated but ephemeral data is lost. |
| 7. Playbook Execution | `terraform force-unlock <lock-id>` (lock release) | N/A - lock release is inherently one-way. The workspace is unlocked and subsequent operations can proceed. | Irreversible. This is why it requires IC approval. |
| 5. Remediation Suggestion | Playbook parameter edit | Revert to agent-suggested defaults (stored separately from engineer edits). | Full. |
| 2. Signal Correlation | Engineer ungroups/regroups alerts | Undo grouping change. Agent re-runs correlation. | Full. |
| 1. Alert Trigger | Alert threshold change | Revert to previous threshold (version history maintained). | Full. |

### Rollback execution

When a playbook step fails:
1. Agent halts execution immediately.
2. Agent presents three options: retry step, skip step (with justification), or initiate rollback.
3. If rollback is selected, agent executes the `rollbackCommand` for each completed step in reverse order.
4. Each rollback step is logged and audited identically to forward execution.
5. After rollback completes, incident returns to Stage 5 (Remediation Suggestion) with the failed execution data available for revised planning.

### Incident reopening

A closed incident can be reopened if:
- New alerts correlate to the same root cause within 24 hours of closure.
- Engineer manually reopens from the incident record.
- Reopening preserves the full history and appends to it. It does not create a new incident.

## Specialist Review Hooks

Infrastructure Architect review is injected at defined points. Review is asynchronous - it does not block the dashboard for other engineers working other incidents.

### Review triggers

| Trigger | What the reviewer sees | SLA |
|---------|----------------------|-----|
| Playbook contains a step with `riskLevel: high` | Full remediation plan, root cause analysis, the specific high-risk step highlighted, and the rollback plan for that step. | 15 minutes for severity-1, 30 minutes for severity-2. |
| Root cause confidence score < 0.6 for the top-ranked cause | Root cause analysis with evidence chain, raw data links, and agent's reasoning log. | 30 minutes. |
| Engineer requests review explicitly | Whatever the engineer selects to share (defaults to full incident context). | Best effort. |

### Review actions

The reviewer (Infrastructure Architect) can:
- **Approve**: Adds sign-off to the approval chain. Execution can proceed.
- **Reject with alternative**: Provides an alternative root cause or remediation approach. Returns to Stage 4 or 5.
- **Request more data**: Asks the agent to gather specific diagnostic data (e.g., "run `terraform state show aws_security_group.main` and show the full attribute map"). Agent executes read-only commands and presents results to reviewer.

### Review does not block

- Other incidents continue to be triaged and worked by other engineers.
- The specific incident awaiting review is in "pending review" state. The engineer can continue investigation (Stages 1-4) but cannot proceed to execution (Stage 7) until review is complete.

## Scope

### In scope

- Run failures and state conflicts on HCP Terraform (org `hashicorp-prod`, project `prod-infrastructure`)
- AI-assisted signal correlation across run logs, audit events, and drift detection results
- Root cause ranking with confidence scores
- YAML-based remediation playbooks with engineer-editable parameters
- RBAC-enforced approval workflow via HCP Terraform Teams (Engineer, Incident Commander, Infrastructure Architect roles)
- Datadog audit trail for all approval and execution actions
- Terraform CLI integration for command execution
- Single-dashboard experience covering all 10 journey stages

### Out of scope

- Module registry issues (separate system, different signal format)
- Provider plugin development (this covers runtime failures, not build-time issues)
- Multi-org correlation (limited to single org `hashicorp-prod` for this iteration)
- Automated remediation without human approval (the human gate is a hard requirement)
- Mobile interface (desktop dashboard only)
- Integration with change management (ServiceNow ITSM integration is a future phase)
- Training the AI model (this PDR covers the inference/execution pipeline, not model development)

## Open Questions

- Datadog log endpoint: is there a dedicated index for incident response audit data, or does it go to the general infrastructure index? Affects query performance for post-incident review. - Ask Datadog admin.
- RBAC enforcement: should the dashboard maintain its own role model or delegate entirely to HCP Terraform Teams? The former is simpler; the latter is more consistent with existing org-level access policy. - Ask IAM team and Derek.
- Runbook versioning: when an engineer edits a playbook's parameters, is the edit persisted as a new playbook version or treated as a one-time override? Versioning enables reuse; one-time overrides are simpler. - Ask Sarah's team.
