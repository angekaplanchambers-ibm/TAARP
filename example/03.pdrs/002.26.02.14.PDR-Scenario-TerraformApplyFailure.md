# PDR-002: Terraform Apply Failure - Scenario

**Status**: Active
**Date**: 2026-02-19
**Author**: Maya Chen
**Depends on**: [PDR-001.Architecture-IncidentResponse](001.26.02.14.PDR-Architecture-IncidentResponse.md)

## Problem Statement

When `terraform apply` on workspace `ws-prod-networking` fails with "Provider produced inconsistent result after apply" on `aws_security_group.main` and the failure correlates with a drift detection result showing manual AWS Console changes to ingress rules,
The Platform Engineer wants to see the correlated signals, confirm the root cause, and execute an approved remediation (refresh state + targeted re-apply),
So that the `ws-prod-networking` workspace is restored to a clean state within the incident SLA without manual cross-referencing of run output across the HCP Terraform UI and AWS Console.

```yaml
name: TFC_DRIFT_RECONCILE_v3
description: Reconcile Terraform state and configuration after drift from out-of-band change
trigger: Provider inconsistency error + drift detection on same resource
version: 3.0
steps:
  - name: Refresh state for target resource
    command: terraform apply -refresh-only -auto-approve -target=${targetResource}
    transport: terraform-cli
    expectedMessage: "Apply complete!"
    risk: low
    rollback: terraform state push terraform.tfstate.backup
  - name: Generate targeted remediation plan
    command: terraform plan -target=${targetResource} -out=${planFile}
    transport: terraform-cli
    expectedMessage: "Plan:"
    risk: low
    rollback: null
  - name: Apply remediation plan
    command: terraform apply ${planFile}
    transport: terraform-cli
    expectedMessage: "Apply complete!"
    risk: high
    review: required
    rollback: terraform state push terraform.tfstate.backup && terraform apply -refresh-only -auto-approve -target=${targetResource}
```

## Open Questions

- The ops-manual IAM user made a direct AWS Console change to a Terraform-managed resource. Should the dashboard integrate with AWS SCPs to proactively prevent out-of-band changes, or is detection-and-remediation the right boundary? - Ask security team.
- Drift detection found the issue at 14:30:02 but the run failure did not occur until 14:32:07. Should drift detection results automatically block queued runs on the affected workspace until an engineer reviews? This would prevent the apply failure entirely but could delay legitimate deployments. - Ask Sarah's team.
- Runbook versioning: when an engineer edits a playbook's parameters, is the edit persisted as a new playbook version or treated as a one-time override? Versioning enables reuse; one-time overrides are simpler. - Ask platform team lead.
