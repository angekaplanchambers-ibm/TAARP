# PDR-003: Incident Response Dashboard - Journey Map Data Contracts

**Status**: Active
**Date**: 2026-02-20
**Author**: Maya Chen
**Depends on**: [PDR-002.Scenario-TerraformApplyFailure](002.26.02.14.PDR-Scenario-TerraformApplyFailure.md)

## Problem Statement

When a Terraform apply failure triggers an incident, the platform engineer needs correlated signals, root cause analysis, and guided remediation - all within a single session. The current workflow requires toggling between run logs, drift detection output, audit trails, and monitoring dashboards manually, constructing the causal chain in their head. Each tool has different data shapes and access patterns. The agent must normalize these into a consistent contract that flows from detection through resolution.

---

## Journey Overview

Ten stages map the lifecycle of incident INC-2026-0847: a Terraform apply failure in workspace ws-prod-networking, caused by an out-of-band manual change to an AWS security group resource.

1. **Alert Trigger** - Run log and drift detection flag the apply failure
2. **Signal Correlation** - Agent groups related signals across run logs, drift reports, and audit trail
3. **Triage View** - Operator sees ranked, correlated incident summary
4. **Root Cause Analysis** - Agent identifies manual out-of-band change as upstream cause
5. **Remediation Suggestion** - Agent proposes state refresh + targeted plan + apply
6. **Approval & Authorization** - Platform Team Lead reviews and approves the plan
7. **Runbook Execution** - Runbook runs remediation steps via Terraform CLI and TFC API
8. **Execution Monitoring** - Real-time step status with rollback triggers
9. **Verification** - Post-remediation health checks confirm resolution
10. **Incident Closure** - Audit record written, incident marked resolved

## Full Journey Map (Data Contracts)

### Stage 1: Alert Trigger

```typescript
interface AlertEvent {
  alertId: string;            // "INC-2026-0847"
  source: 'TFC-RUN-LOG' | 'TFC-AUDIT' | 'TFC-DRIFT' | 'WEBHOOK';
  workspace: string;          // "ws-prod-networking"
  runId?: string;             // "run-abc123"
  severity: 'critical' | 'high' | 'medium' | 'low';
  timestamp: string;          // ISO 8601: "2026-02-20T09:14:32.000Z"
  rawMessage: string;         // "Error: Provider produced inconsistent result after apply for aws_security_group.main"
  correlationKey: string;     // "CK-20260220-091432-ws-prod-networking"
}
```

**Source systems**: HCP Terraform run log stream (via TFC API), drift detection (via TFC scheduled runs)
**Trigger condition**: Severity >= high, or any apply failure or drift detection event

### Stage 2: Signal Correlation

### Happy Path

```
detected -> correlating -> triaging -> analyzing -> remediation_suggested
-> pending_approval -> approved -> executing -> verifying -> resolved
```

### Error Path 1: Escalation (approval timeout)

```
... -> pending_approval -> (15 min timeout) -> escalated
```

The `escalated` state pages the Infrastructure Architect. A new approval request is generated with `approverRole: 'infrastructure-architect'`. If approved, the incident transitions to `approved -> executing`.

### Error Path 2: Execution failure with rollback

```
... -> executing -> failed -> rolling_back -> remediation_suggested
```

When a runbook step fails (non-zero return code), the engine:
1. Marks the execution as `failed`
2. Runs rollback commands for all completed steps in reverse order
3. Transitions back to `remediation_suggested` so the agent can propose an alternative plan
4. If rollback itself fails, transitions to `escalated`

### Error Path 3: Verification failure

```
... -> verifying -> (checks fail) -> analyzing
```

If verification checks fail, the incident returns to `analyzing` for the agent to re-evaluate root cause with new data from the failed remediation attempt.

## Timing Budget

| Stage | Wall Clock | Agent Processing | Notes |
|-------|-----------|-----------------|-------|
| 1. Alert Trigger | 0-5s | <1s | Depends on webhook delivery or polling interval |
| 2. Signal Correlation | 5-15s | 3-8s | ML model inference + run log and audit trail queries |
| 3. Triage View | 2-10s | <1s | Operator reads and selects incident |
| 4. Root Cause Analysis | 5-20s | 3-12s | Pattern matching against historical data |
| 5. Remediation Suggestion | 3-10s | 2-5s | Runbook lookup + parameter substitution |
| 6. Approval & Authorization | 30s-15min | <1s | Human decision time dominates |
| 7. Runbook Execution | 15-60s | 15-60s | Sequential command execution |
| 8. Execution Monitoring | (concurrent with 7) | <1s per poll | 2s polling interval |
| 9. Verification | 30-120s | 5-15s | Checks run + 60s quiet window |
| 10. Incident Closure | 5-10s | 2-5s | Audit write + state update |

**Total happy path**: 4-8 minutes (approval within 30s)
**Total with escalation**: 15-30 minutes (approval timeout at 15min, re-approval needed)
**Total with execution failure + rollback**: 10-20 minutes (rollback + re-analysis + second execution)

## Bridging Data Contract

These values remain consistent across all stages of an incident. Every service call and every UI render references this shared identity.

```typescript
interface IncidentBridge {
  incidentId: string;         // "INC-2026-0847" - assigned at stage 1
  correlationKey: string;     // "CK-20260220-091432-ws-prod-networking" - assigned at stage 2
  workspace: string;          // "ws-prod-networking" - from initial alert
  resource: string;           // "aws_security_group.main" - from correlation
  operatorId: string;         // "op-mchen" - assigned at triage
  approvalToken?: string;     // set after stage 6, required for stages 7-10
  executionId?: string;       // set at stage 7, required for stages 8-10
}
```

**Consistency rules**:
- `incidentId` is immutable after creation
- `correlationKey` is immutable after stage 2 completes
- `operatorId` can change only via explicit reassignment (logs old and new)
- `approvalToken` expires after 15 minutes or after execution completes
- `executionId` changes if a second execution attempt is needed (after rollback)
