# 002. User Journey Map - Terraform Remediation Agent

**Date**: 2026-02-17
**Author**: Maya Chen
**Status**: Active
**Depends on**: [001.IncidentResponse-Strategy](001.26.02.14.IncidentResponse-Strategy.md)

**Scope**: Detection-through-resolution flow for HCP Terraform workspace failures and drift incidents, with AI-assisted triage and operator-approved remediation.

## Personas

| Persona | Role | Category | Mental Model |
|---------|------|----------|--------------|
| Sarah (Platform Engineer) | Monitors Terraform workspaces, investigates run failures and drift | User | "I watch workspace runs and react when something goes red" |
| Derek (Platform Team Lead) | Decision maker during major infrastructure incidents | Buyer | "I need enough context to approve or escalate, fast" |
| James (Infrastructure Architect) | Deep-dive specialist for state and provider issues | Champion | "I need the raw plan output and state data to diagnose root cause" |

### Platform Engineer (Sarah's team)

**Category**: User
**Quote**: "I'm watching twelve production workspaces across three cloud accounts. By the time I spot the pattern across all of them, we've already lost ten minutes."

**Context**:
- **Role**: First responder for infrastructure health. Owns initial triage and escalation. Hands off to Infrastructure Architect for deep diagnosis or to Platform Team Lead for approval of high-impact actions.
- **Environment**: HCP Terraform UI, terraform CLI, AWS console, GCP console, Datadog dashboards. Typically 3-5 browser tabs and a terminal open simultaneously.
- **Frequency**: Monitors continuously during shift (8-hour rotation). Handles 3-5 incidents per week, 1-2 requiring escalation.

**Goals**:
- See correlated signals across workspaces and providers in one view instead of scanning run logs individually
- Reduce time from alert to correct triage decision
- Know when to escalate vs. handle independently

**Pain Points**:
- Run logs are verbose and unsorted. Critical errors get buried in provider output.
- Correlation between workspace failures and upstream drift requires manual cross-referencing of timestamps and resource addresses.
- No structured way to capture triage reasoning for handoff.

**Design Implications**:
- Dashboard must surface pre-correlated signals, not raw run logs
- Triage view needs to show the agent's correlation reasoning so the engineer can verify it
- Escalation must be a single action with context automatically attached

### Platform Team Lead (Derek)

**Category**: Buyer
**Quote**: "I don't need to understand the provider internals. I need to know: what failed, what's the proposed fix, what's the blast radius, and who reviewed it."

**Context**:
- **Role**: Approves or rejects remediation actions during major incidents. Owns the incident record. Reports to management on resolution timelines.
- **Environment**: Incident management tool (PagerDuty), Slack, email. Joins the dashboard during active incidents. Does not use terraform CLI directly.
- **Frequency**: Engaged for severity-1 and severity-2 incidents. 2-3 per month.

**Goals**:
- Approve or reject remediation with full context visible in one screen
- Maintain audit trail for post-incident review
- Keep resolution time under SLA thresholds

**Pain Points**:
- Context arrives piecemeal via Slack and email during incidents. No single view.
- Approval is currently verbal or via Slack message, with no structured record.
- Post-incident reviews reveal gaps in what was communicated during the incident.

**Design Implications**:
- Approval screen must show: incident summary, proposed action, blast radius, who recommended it, rollback plan
- Approval action must capture reason field and write to Datadog audit trail
- RBAC enforcement - only Platform Team Lead role can approve high-impact remediations

### Infrastructure Architect (James)

**Category**: Champion
**Quote**: "The dashboard is fine for triage, but when I'm called in, I need to see the raw plan JSON and the actual state file. Don't summarize away the details I need."

**Context**:
- **Role**: Deep technical specialist. Called in when the engineer cannot resolve or when the issue involves state-level problems (state drift, provider bugs, resource import conflicts).
- **Environment**: terraform CLI, plan JSON output, state inspection commands, provider debug logs, Datadog log explorer. VS Code with HCL extensions.
- **Frequency**: Engaged for 20-30% of incidents. Spends 1-4 hours per engagement.

**Goals**:
- Drill from dashboard summary into raw plan and state data without switching tools
- Verify or override the agent's root cause analysis with direct evidence
- Run ad-hoc terraform commands from the dashboard context

**Pain Points**:
- Current tools require switching between HCP Terraform UI, terminal, and cloud consoles to build a complete picture.
- Agent summaries that hide raw data force him to re-derive the same conclusions independently.
- No way to annotate findings back into the incident record from diagnostic tools.

**Design Implications**:
- Every agent-generated summary must link to the underlying raw data
- Dashboard must support drill-down to raw plan JSON, state file diffs, and provider error logs
- "Open in terminal" action to launch a terraform CLI session with pre-populated context

## Jobs To Be Done

When a Terraform workspace failure or drift event is detected across production workspaces,
I want to see AI-correlated signals, triage the root cause, and execute an approved remediation,
So that mean time to resolution stays under SLA without requiring manual cross-referencing of run logs.

- **Functional**: Correlate alerts from run failures, drift detection, and audit events into a ranked root cause analysis. Execute remediation runbooks with operator approval.
- **Emotional**: Confidence that the agent's correlation is trustworthy and that approved actions are reversible.
- **Social**: Demonstrate to management and audit that incident response followed a structured, reviewable process.

## Per-Stage Detail

### Stage 1: Alert Trigger
**User Action**: Engineer sees alert notification in dashboard alert panel. Clicks to expand.
**Agent Action**: Ingests HCP Terraform run failure events and drift detection results. Generates initial alert with severity classification based on error type and workspace tier (production, staging).
**Touchpoint**: Dashboard alert panel - card with severity badge, workspace name, timestamp, error excerpt.
**System Calls**:
- Run status: `GET /api/v2/runs/{run_id}` (check run status: errored, canceled, policy_soft_failed)
- Webhook payload: HCP Terraform notification webhook with run event data
- Audit trail: `GET /api/v2/organization/audit-trail` (filtered by recent events)
**Data In**: Run failure events, webhook payloads, workspace drift detection results
**Data Out**: `AlertEvent { alertId, source, workspaceId, workspaceName, severity, errorType, errorMessage, runId, timestamp, rawLogRef }`
**Validation**: Alert deduplication - suppress identical alerts within a 60-second window. Severity mapping verified against workspace tier and error classification table.
**Failure Mode**: Webhook delivery delay (common: HCP Terraform retries on failure). Fallback: poll `GET /api/v2/workspaces/{id}/runs?filter[status]=errored` on 30-second interval.
**Duration**: < 5 seconds from webhook delivery to dashboard notification.

### Stage 2: Signal Correlation
**User Action**: Engineer reviews the correlated signal set presented on a timeline. Verifies that the grouped alerts belong to the same incident.
**Agent Action**: Correlates alerts by time window (120-second default), resource address (e.g., `aws_security_group.main`), and workspace dependency chain. Groups into a single incident envelope. Assigns correlation confidence score.
**Touchpoint**: Correlation timeline view - horizontal timeline with grouped alert markers, confidence badge.
**System Calls**:
- Recent failures: `GET /api/v2/workspaces/{id}/runs?filter[status]=errored` (last 24 hours of workspace run history)
- Drift results: `GET /api/v2/workspaces/{id}/health` (drift detection status and affected resources)
- Cross-reference: agent queries internal correlation rules engine (local, no external call)
**Data In**: Multiple `AlertEvent` objects from Stage 1
**Data Out**: `IncidentEnvelope { incidentId, alerts[], correlationScore, timeWindow, affectedResources[], correlationReason }`
**Validation**: Correlation score > 0.7 required to auto-group. Scores 0.4-0.7 presented as "suggested" grouping for engineer confirmation.
**Failure Mode**: False correlation (grouping unrelated alerts). Engineer can ungroup signals manually. Agent learns from correction.
**Duration**: < 10 seconds after final alert in the correlation window.

### Stage 3: Triage View
**User Action**: Engineer selects the incident from the queue. Reads the triage summary: affected workspaces, signal count, severity, recommended next step.
**Agent Action**: Generates triage summary from the incident envelope. Highlights the highest-severity signal. Suggests whether this is engineer-resolvable or requires escalation based on incident type and historical patterns.
**Touchpoint**: Triage summary panel - incident header, affected resource list, signal timeline, escalation recommendation.
**System Calls**:
- Historical runs: `GET /api/v2/runs?filter[workspace][name]=ws-prod-networking&filter[status]=errored&page[size]=20` (recent failure history for workspace)
- Workspace detail: `GET /api/v2/workspaces/{id}?include=current-run` (current workspace state)
**Data In**: `IncidentEnvelope` from Stage 2, historical run data
**Data Out**: `TriageSummary { incidentId, severity, affectedWorkspaces[], signalCount, escalationRecommendation, suggestedNextStep, historicalSimilar[] }`
**Validation**: Escalation recommendation must include reasoning (e.g., "Similar incident INC-2025-0412 required Infrastructure Architect intervention").
**Failure Mode**: Historical data unavailable. Agent proceeds without historical context, flags "no historical match" in summary.
**Duration**: < 3 seconds to render triage view after incident selection.

### Stage 4: Root Cause Analysis
**User Action**: Engineer reviews ranked root causes. Clicks into each cause to see supporting evidence. Infrastructure Architect (if engaged) drills into raw plan JSON and state data.
**Agent Action**: Runs root cause analysis using correlated signals. Produces ranked list of probable causes with confidence scores and evidence links. Each cause references specific run IDs, resource addresses, and provider error messages.
**Touchpoint**: Root cause detail panel with ranked cards. Each card expands to show evidence chain. "View raw data" link per evidence item.
**System Calls**:
- State inspection: `terraform state show aws_security_group.main` (current resource state)
- Targeted plan: `terraform plan -json -target=aws_security_group.main` (diff for specific resource)
- State version: `GET /api/v2/state-versions/{id}` (full state file from HCP Terraform)
- Plan JSON output: `GET /api/v2/plans/{id}/json-output` (structured plan data)
**Data In**: `IncidentEnvelope`, state data, plan JSON, provider error logs
**Data Out**: `RootCauseAnalysis { incidentId, causes[{ rank, description, confidence, evidenceChain[], rawDataRefs[] }] }`
**Validation**: Each cause must cite at least one raw data reference. Confidence scores must sum to <= 1.0 across ranked causes.
**Failure Mode**: State file locked by an active run. Agent returns "analysis incomplete" with available evidence and re-runs after a configurable delay.
**Duration**: 10-30 seconds depending on plan execution time and state file size.

### Stage 5: Remediation Suggestion
**User Action**: Engineer reviews the suggested remediation runbook. Edits parameters if needed (e.g., target resource, refresh-only flag). Infrastructure Architect reviews technical accuracy for high-risk actions.
**Agent Action**: Selects remediation runbook from library based on root cause. Populates runbook parameters from incident context. Flags steps that require specialist review.
**Touchpoint**: Runbook preview panel - step-by-step view with editable parameter fields. Risk badge per step. "Requires review" flag on high-risk steps.
**System Calls**:
- Runbook lookup: `GET /api/runbooks?rootCause=DRIFT_APPLY_CONFLICT`
- Parameter population: agent fills from incident context (no external call)
**Data In**: `RootCauseAnalysis`, runbook library (YAML), incident context
**Data Out**: `RemediationPlan { incidentId, runbook, steps[{ order, command, parameters, riskLevel, requiresReview, rollbackCommand }], estimatedDuration }`
**Validation**: Every step with `riskLevel: high` must have `requiresReview: true`. Every mutating step must have a `rollbackCommand`.
**Failure Mode**: No matching runbook. Agent suggests manual resolution steps based on root cause description and flags for engineer to build ad-hoc runbook.
**Duration**: < 5 seconds for runbook selection and parameter population.

### Stage 6: Approval & Authorization
**User Action**: Engineer submits remediation for approval. Platform Team Lead reviews the plan, blast radius, and rollback steps. Approves or rejects with a reason. For high-risk steps, Infrastructure Architect provides technical sign-off first.
**Agent Action**: Routes approval request based on RBAC rules. Packages incident summary, remediation plan, blast radius assessment, and rollback plan into approval payload. Records approval decision to Datadog audit trail.
**Touchpoint**: Approval dialog with structured fields: incident summary, proposed actions, blast radius, rollback plan, reason field (required).
**System Calls**:
- RBAC check: `GET /api/v2/teams/{id}/users` (verify approver has correct team membership)
- Audit write: `POST /api/audit/events` (Datadog Log Management endpoint)
- Notification: webhook to Slack channel `#incident-{incidentId}` via notification configuration
**Data In**: `RemediationPlan`, RBAC policy, incident context
**Data Out**: `ApprovalRecord { incidentId, approver, decision, reason, timestamp, auditTrailRef }`
**Validation**: Rejection requires reason (min 20 characters). Approval for severity-1 requires Platform Team Lead role. Technical sign-off required for steps flagged `requiresReview`.
**Failure Mode**: Approver unavailable. Escalation to backup approver after configurable timeout (default: 15 minutes). Agent sends reminder at 10-minute mark.
**Duration**: Variable. Target: < 5 minutes for severity-1. Measured from request to decision.

### Stage 7: Runbook Execution
**User Action**: Engineer initiates execution after approval. Watches step-by-step progress.
**Agent Action**: Executes runbook steps sequentially. Each step: run command, capture output, validate result, proceed or halt. On failure: stop execution, preserve state, notify engineer.
**Touchpoint**: Execution console - step list with status indicators (pending, running, success, failed). Output panel per step.
**System Calls** (example for drift remediation on a security group):
- Step 1: `terraform apply -refresh-only -auto-approve -target=aws_security_group.main` (sync state with actual infrastructure)
- Step 2: `terraform plan -target=aws_security_group.main -out=remediation.tfplan` (generate targeted remediation plan)
- Step 3: `terraform apply remediation.tfplan` (apply the remediation)
- Audit: `POST /api/audit/events` per step completion
**Data In**: `ApprovalRecord`, `RemediationPlan`
**Data Out**: `ExecutionLog { incidentId, steps[{ order, command, output, status, duration }], overallStatus }`
**Validation**: Each step's output checked against expected pattern (e.g., `Apply complete! Resources: 0 added, 1 changed, 0 destroyed.`). Unexpected output halts execution.
**Failure Mode**: Step fails or times out. Agent halts, preserves execution state, notifies engineer. Engineer can: retry step, skip step (with justification), or initiate rollback.
**Duration**: 30 seconds to 5 minutes depending on step count and provider response.

### Stage 8: Execution Monitoring
**User Action**: Engineer monitors execution progress in real time. Platform Team Lead watches status dashboard for escalation triggers.
**Agent Action**: Streams execution output. Updates step status in real time. Calculates estimated time remaining. Detects anomalies in execution output (unexpected errors, abnormal timing).
**Touchpoint**: Progress monitor - real-time step status, output stream, ETA, anomaly alerts.
**System Calls**:
- WebSocket for real-time updates: `wss://{dashboard}/ws/execution/{incidentId}`
- Apply log streaming: `GET /api/v2/applies/{id}/log` (SSE streaming of apply output from HCP Terraform)
**Data In**: `ExecutionLog` (streaming), workspace health data
**Data Out**: `MonitoringState { incidentId, currentStep, overallProgress, eta, anomalies[] }`
**Validation**: If execution duration exceeds 2x estimated, agent flags anomaly and suggests engineer review.
**Failure Mode**: WebSocket disconnection. Fallback to polling at 5-second intervals. Execution continues server-side regardless of dashboard connection.
**Duration**: Matches Stage 7 duration. Monitoring overhead: < 1 second per status update.

### Stage 9: Verification
**User Action**: Engineer runs verification checks from the checklist. Infrastructure Architect (if engaged) runs independent diagnostics. Results feed back into the incident record.
**Agent Action**: Generates verification checklist based on root cause and remediation steps. Auto-runs verifiable checks (e.g., clean plan, workspace health status). Flags manual checks for engineer.
**Touchpoint**: Verification checklist - automated checks with pass/fail, manual checks with confirmation buttons.
**System Calls**:
- Clean plan check: `terraform plan` (expect output: "No changes. Your infrastructure matches the configuration.")
- Workspace status: `GET /api/v2/workspaces/{id}?include=current-run` (verify workspace shows healthy state)
- Drift detection: trigger a drift detection re-run via HCP Terraform to confirm no remaining drift
- Provider health: verify no new errored runs in the past 5 minutes via `GET /api/v2/workspaces/{id}/runs?filter[status]=errored&page[size]=5`
**Data In**: `ExecutionLog`, current workspace state
**Data Out**: `VerificationResult { incidentId, checks[{ name, type, status, detail }], overallStatus, verifiedBy }`
**Validation**: All automated checks must pass. Manual checks require engineer confirmation. If any check fails, incident remains open.
**Failure Mode**: Verification reveals remediation did not resolve the issue. Incident returns to Stage 4 (Root Cause Analysis) with additional data from the failed remediation.
**Duration**: 2-5 minutes for automated checks. Manual checks are engineer-paced.

### Stage 10: Incident Closure
**User Action**: Engineer marks incident resolved. Adds resolution notes. Platform Team Lead signs off on the incident record.
**Agent Action**: Generates incident summary with timeline, root cause, remediation steps taken, and verification results. Writes closure record to Datadog audit log. Archives incident data.
**Touchpoint**: Incident closure form - pre-filled summary (editable), resolution category dropdown, lessons learned field, sign-off button.
**System Calls**:
- Incident record write: `POST /api/incidents/{id}/close`
- Audit write: `POST /api/audit/events` (Datadog closure event with full incident timeline)
- Archive: `PUT /api/incidents/{id}/archive`
**Data In**: All prior stage outputs, engineer notes
**Data Out**: `IncidentRecord { incidentId, timeline[], rootCause, remediationSteps[], verificationResult, closedBy, closedAt, resolutionCategory, lessonsLearned }`
**Validation**: Closure requires: all verification checks passed, lead sign-off (for severity-1/2), resolution category selected.
**Failure Mode**: Lead sign-off delayed. Incident remains in "pending closure" state. Agent sends daily reminder until signed off or escalated.
**Duration**: 2-5 minutes for engineer, async for lead sign-off.
