## What This Doc Is

A strategy document for the Incident Response Dashboard - an AI-assisted triage and remediation tool for platform engineers managing HCP Terraform infrastructure. This document synthesizes the kickoff (2026-02-10) and technical deep dive (2026-02-14) into scope boundaries, personas, critical user journeys, and a deliverables plan.

This doc does not define the UI layout, technical architecture, or data model. Those are downstream artifacts (journey maps and PDRs) that trace back to the scope and CUJs defined here.

## The Use Case

Platform engineers managing 47 Terraform workspaces across 3 projects spend an average of 38 minutes resolving P1/P2 run failure and drift incidents. Of that time, roughly 30 minutes is information gathering - opening multiple tools (HCP Terraform UI, AWS console, Datadog, Slack), manually correlating timestamps across structured and unstructured data sources, and mentally matching signal patterns to known incident types.

These patterns are well-understood by senior engineers but live as tribal knowledge. Junior and off-hours engineers perform the same diagnostic sequences more slowly and less confidently, often escalating via Slack to senior staff.

The dashboard ingests HCP Terraform audit log events, run log API data, and notification webhooks. An anomaly detection model correlates these signals in near-real-time, presents a grouped incident view with root cause hypotheses, and offers remediation playbooks that engineers can approve and execute from the dashboard.

The target outcome is a 50% MTTR reduction (from 38 minutes to under 19 minutes) for run failure and drift P1/P2 incidents, measured over the first 3 months of production use.

## Scope

### In Scope

- **Incident types**: Run failures, state lock conflicts, drift detection violations, Sentinel policy failures, and cross-workspace cascade failures. These represent approximately 65% of P1/P2 incident volume per Sarah Kim's 6-month PagerDuty export. [Kickoff, D3]
- **Workflow boundary**: Detection through resolution. Starts when signals indicate an anomaly, ends when the engineer confirms the system has recovered. [Kickoff, D1]
- **AI role**: Suggest root causes with confidence scores and reasoning chains. Suggest remediation steps from pre-defined playbooks. All suggestions require explicit engineer approval. [Kickoff, D2]
- **Remediation execution**: YAML-defined playbooks executed via terraform CLI through the HCP Terraform API. Playbooks cover workspace force-unlock, run retry, targeted apply, state rollback, Sentinel override request, and combined cascade recovery. [Deep Dive, D2]
- **Audit**: Immutable append-only log of all AI suggestions, engineer actions, and remediation step results. Replicated to Datadog via the Datadog integration. [Deep Dive, D4]
- **RBAC**: Three tiers - observe, approve-standard, approve-critical - mapped to existing operational hierarchy. [Deep Dive, D5]

### Out of Scope

- **Prevention and prediction**: Capacity trend analysis, cost anomaly detection, and proactive alerting are separate initiatives with different data requirements. [Kickoff, D1]
- **Module registry issues, provider plugin bugs, workspace variable conflicts**: These are v2 candidates with different signal characteristics. [Kickoff, D3]
- **Autonomous remediation**: No action executes without engineer approval. [Kickoff, D2]
- **Alert routing**: Integration with PagerDuty for initial alerting is an open question. The dashboard is the triage surface, not a replacement for the alerting pipeline.
- **Model training infrastructure**: MLOps pipeline for retraining the anomaly detection model is a separate workstream.

## Critical User Journeys

### CUJ-1 (Primary): AI-Correlated Incident Triage

> As a **Platform Engineer**, I want to see AI-correlated run failure signals in one view so I can triage faster than switching between HCP Terraform UI, cloud consoles, and Slack.

**Trigger**: Anomaly detection service groups correlated signals and creates an incident record.

**Core flow**:
1. Engineer sees new incident on the dashboard with severity, affected workspaces (prod-networking, prod-compute), and incident type classification
2. Engineer opens the incident detail - a signal timeline showing correlated audit log anomalies, run log error patterns, and webhook notifications on a shared time axis
3. AI-generated summary describes the probable root cause (e.g., "State lock conflict on prod-networking workspace causing cascading plan failures in dependent workspaces prod-compute and prod-storage") with a confidence score and the reasoning chain showing which signals led to this conclusion
4. AI also shows what it ruled out (provider rate limiting, credential expiry, network connectivity issues)
5. Engineer evaluates the suggestion, checks the reasoning, and either accepts or investigates further

**Success criteria**: Engineer reaches a triage decision (accept AI suggestion, investigate further, or escalate) in under 5 minutes. Current baseline for the same decision point is 20-30 minutes.

**Grounded in**: Kickoff discussion of the prod-networking incident that took 47 minutes; Sarah's description of the manual correlation sequence; Maya's engineer interviews on reasoning transparency.

### CUJ-2 (Supporting): Remediation Approval and Execution

> As a **Platform Team Lead**, I want to approve agent-suggested remediations with full context so I can make informed decisions about production changes.

**Trigger**: AI suggests a remediation playbook for an active incident, or an engineer requests approval for a critical remediation.

**Core flow**:
1. Platform Team Lead sees a pending approval notification with the incident summary, AI's root cause analysis, and the specific playbook being proposed
2. Reviews the playbook steps - each step shows the terraform CLI command that will execute, the expected outcome, and the rollback action if it fails
3. Reviews preconditions - the dashboard shows whether each precondition is currently met (e.g., no other runs queued in workspace, state serial matches expected value)
4. Approves with a mandatory reason field ("State lock conflict confirmed via audit log at 14:22:03, approving force-unlock on prod-networking workspace")
5. Monitors step-by-step execution with real-time status updates
6. Confirms system recovery via the same signal timeline that showed the anomaly

**Success criteria**: Platform Team Lead has enough context to approve or reject without independently diagnosing the incident. Approval decision takes under 3 minutes.

**Grounded in**: Kickoff D2 (engineer approves all remediations); Deep Dive discussion of RBAC tiers and the three-tier approval model; Deep Dive D3 (mandatory reason field).

### CUJ-3 (Supporting): Raw Signal Deep Dive

> As an **Infrastructure Architect**, I want to drill into raw run logs, state diffs, and provider debug output from the dashboard so I can validate the AI's root cause analysis.

**Trigger**: Escalation from an engineer or platform team lead when the AI's suggestion doesn't match observations, or when the incident involves an unusual pattern.

**Core flow**:
1. Infrastructure Architect opens the incident and sees the AI summary and reasoning chain
2. Clicks into the reasoning chain to see the specific data points - individual audit log entries, specific run log error lines, exact webhook payload content with timestamps
3. Applies filters and pivots on the raw data - filter by workspace, by run ID, by Terraform resource address, by time window
4. Identifies a signal the AI missed or misinterpreted (e.g., the state lock conflict is a symptom of a provider API timeout in a different workspace's apply phase)
5. Provides corrective input that feeds back into the incident record and the AI's feedback loop

**Success criteria**: Infrastructure Architect can access raw signal data for any AI conclusion without leaving the dashboard. Drill-down from AI summary to raw record takes under 30 seconds.

**Grounded in**: Sarah's request in kickoff to show "what the AI ruled out"; James's description of signal fingerprints in the deep dive; Maya's interview finding that engineers want to see the work behind the AI's conclusion.

## Key Technical Context

The following technical decisions constrain the design. These are settled and come from the technical deep dive.

**Data pipeline latency**: Audit log events arrive via polling (`GET /api/v2/organization/audit-trail`) at 30-second intervals. Run logs arrive via SSE streaming (`GET /api/v2/plans/{id}/log` and `GET /api/v2/applies/{id}/log`). Webhook notifications arrive in near-real-time for run failures and policy violations. The UI should not promise real-time if the underlying polling data has a 30-second lag. [Deep Dive, James Liu]

**Remediation execution model**: All playbooks execute via terraform CLI authenticated with an HCP Terraform API token. Each step has a timeout and a rollback command. Playbooks support conditional branching (e.g., retry vs. force-unlock vs. state rollback). The dashboard does not execute arbitrary shell commands or interact with cloud provider APIs directly. [Deep Dive, D2 and Alex Torres]

**RBAC enforcement**: Two layers - the dashboard checks the user's role via HCP Terraform Teams before sending a command, and Sentinel policies enforce guardrails at the workspace level. The dashboard RBAC is not a replacement for Sentinel; it's an additional gate. [Deep Dive, D5]

**Audit immutability**: The audit trail is append-only and immutable via the Datadog integration. This is a compliance requirement, not a preference. The dashboard cannot modify or redact audit entries. [Deep Dive, D4]

**Deployment target**: Anomaly detection model will likely run on standard cloud compute (ECS) with network isolation to avoid shipping sensitive state data across trust boundaries. James Liu is benchmarking inference performance. This may constrain model complexity. [Deep Dive, open question]

## Open Questions

1. **Cross-workspace incident model**: When a state lock conflict in one workspace causes cascading plan failures in dependent workspaces, is that one incident with multiple signals or two linked incidents? This affects the data model, the UI grouping, and the AI's correlation logic. Need input from Sarah and James. [Raised in Kickoff]

2. **PagerDuty integration**: Does the dashboard surface incidents directly (push model via WebSocket) or does PagerDuty remain the alerting entry point with the dashboard as a drill-down destination? Affects CUJ-1's trigger definition. [Raised in Kickoff]

3. **Model hosting**: Cloud compute with network isolation vs. on-prem for sensitive state data. James is benchmarking inference performance on ECS. This decision affects data pipeline architecture and model complexity constraints. [Raised in Deep Dive]

4. **Engineer feedback mechanism**: How does the engineer mark an AI suggestion as helpful, unhelpful, or partially correct? The weekly retrain cycle needs structured feedback, not just the implicit signal of "engineer diverged from suggestion." [Derived from Deep Dive audit discussion]

5. **Off-hours critical approval**: During nights and weekends, only one senior engineer may be on shift. If a critical remediation is needed and no approve-critical role is available, what's the escalation path? Phone call to Derek or James, or a time-limited override? [Raised in Deep Dive]

6. **Webhook reliability during cascade failures**: James flagged that webhook delivery may be delayed or dropped during high-volume cascade events. If the AI depends on webhooks as a primary signal, delivery failures during cascade incidents could create a gap in exactly the scenario where full signal coverage matters most. Pending James's review of webhook retry policies and fallback to polling. [Raised in Deep Dive]
