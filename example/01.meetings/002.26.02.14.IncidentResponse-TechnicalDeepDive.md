# 02.2026-02-14-IncidentResponse-TechnicalDeepDive

## TL;DR

The data pipeline for AI-assisted Terraform incident triage is feasible using HCP Terraform's existing APIs. Primary signals are the audit log (every API call and run state change) and run logs (plan/apply output), both accessible via REST endpoints with near-real-time latency. Drift detection results and notification webhooks provide supplementary signals. Remediation runbooks will be YAML-defined and executed via the Terraform CLI. All remediations require engineer approval with a mandatory reason field. The audit log is immutable and shipped to Datadog. RBAC will use three tiers: observe, approve-standard, and approve-critical.

## Context

Follow-up to the kickoff meeting on 2026-02-10. This session covers the technical integration points, data sources, anomaly detection requirements, and remediation execution model. James Liu (Infrastructure Architect) and Alex Torres (Automation Engineer) joined to address questions about API availability, rate limits, and automation constraints that came up in the kickoff.

## Key Discussion Points

### Data Sources and Signal Pipeline

James walked through the available data sources from HCP Terraform:

**HCP Terraform Audit Log**
- Every API call, run state change, and policy check generates an audit event
- Accessible via `GET /api/v2/organization/audit-trail` with pagination and filtering
- Roughly 50,000 events per day across the organization's 140 workspaces
- Contains actor identity, action type, resource affected, timestamp, and request metadata
- James noted the API rate limit of 30 requests per second per organization - the ingestion pipeline needs to respect backoff headers

**Run Logs**
- Plan and apply output for every Terraform run, accessible via `GET /api/v2/plans/{id}/log` and `GET /api/v2/applies/{id}/log`
- Real-time streaming available via Server-Sent Events (SSE) during active runs
- Contains provider error messages, resource diffs, state change details, and timing information
- The three highest-volume workspaces (networking-prod, compute-prod, and data-prod) generate roughly 120 runs per day combined

**Drift Detection Results**
- Periodic health assessments via `GET /api/v2/workspaces/{id}/health`
- Runs every 24 hours per workspace by default, configurable down to every 2 hours
- James can enable more frequent drift checks on critical workspaces, but this consumes run quota - the organization plan allows 50 concurrent runs

**Notification Webhooks**
- Run failures, policy violations, and drift detection results pushed via webhook
- Near-real-time delivery, configured per workspace
- Webhook payloads include run ID, workspace ID, status, and a link to the full run details

**Sentinel Policy Results**
- Accessible via `GET /api/v2/policy-checks/{id}`
- Returns hard-mandatory, soft-mandatory, and advisory results with detailed rule evaluations
- Useful for catching policy violations before they become incidents

**Proposed Signal Pipeline**
- Audit log: polled via REST API every 30 seconds, respecting rate limits
- Run logs: streamed via SSE for active runs, polled for historical data
- Drift detection: polled every 2 hours for critical workspaces, daily for others
- Notification webhooks: pushed to an ingestion endpoint in real-time
- All four converge in the anomaly detection service, which correlates by timestamp and workspace ID

Sarah asked about the 30-second polling latency for the audit log. James said it can be reduced by switching to a webhook-first model where the audit log poll fills in anything the webhook missed, but this requires deploying a publicly reachable webhook endpoint with proper TLS and authentication. The team agreed to start with 30-second polling and add webhooks for critical workspaces if latency becomes a problem.

### Anomaly Detection Model Requirements

Derek framed the question: what does the model need to detect, and what training data exists?

Sarah provided the incident categorization from her PagerDuty export:

| Incident Type | Monthly Avg (P1/P2) | Pattern Recognizable? |
|---|---|---|
| Apply failure (provider error) | 4.1 | Yes - error string parsing + state diff analysis |
| State lock conflict | 2.3 | Yes - lock metadata + concurrent run detection |
| Drift detected (out-of-band change) | 3.7 | Yes - drift run output + audit log correlation |
| Sentinel policy violation | 1.8 | Yes - policy check result + change set analysis |
| Plan timeout / agent exhaustion | 0.9 | Partially - requires run duration + agent pool metrics |
| Cross-workspace cascade failure | 1.4 | Yes - run trigger chain + shared module analysis |

James added that each of these has a distinct signal fingerprint:
- Apply failure: provider error string in the apply log (e.g., `Error: creating EC2 Instance`, `Error: Provider produced inconsistent result`), correlated with resource diff showing the failed change
- State lock conflict: `Error: Error locking state` in the run log plus concurrent active runs in the same workspace visible in the audit log
- Drift detected: health check returning `drifted` status plus audit log entries showing API calls or console actions from outside Terraform

Alex noted that the model needs a time-windowed correlation engine, not just pattern matching on individual signals. A single apply failure is routine. An apply failure plus a state lock conflict in the same workspace within a 10-minute window plus a policy violation on a dependent workspace - that's a cascade pattern requiring coordinated remediation.

### Remediation Runbook Format

Alex presented the runbook format his team has been developing for non-AI automation:

```yaml
runbook: drift-remediation
version: 1.0
trigger: ai-suggested
severity: standard
target:
  type: terraform-workspace
  organization: ${ORG_ID}
  workspace: ${WORKSPACE_ID}
preconditions:
  - check: workspace-unlocked
    expect: true
  - check: no-active-runs
    expect: true
steps:
  - action: refresh-state
    command: "terraform apply -refresh-only -auto-approve -target=${RESOURCE_ADDRESS}"
    timeout: 120s
    rollback: "terraform state push ${PRIOR_STATE_VERSION}"
  - action: generate-plan
    command: "terraform plan -target=${RESOURCE_ADDRESS} -out=remediation.tfplan"
    timeout: 300s
  - action: apply-plan
    command: "terraform apply remediation.tfplan"
    timeout: 600s
    rollback: "terraform apply -destroy -target=${RESOURCE_ADDRESS}"
approval:
  required: true
  tier: standard
  reason_required: true
audit:
  log_to: datadog
  include: [engineer, timestamp, reason, step_results]
```

Derek asked about the Terraform CLI dependency. Alex confirmed that Terraform CLI 1.7+ is deployed on the automation agents, authenticated via TFE tokens scoped to specific workspaces. The commands run through HCP Terraform's API-driven runs or local execution mode depending on the workspace configuration, with all actions recorded in the audit log.

James raised a concern about the `apply` step - applying targeted resources can have unintended effects on dependent resources. For some incidents, a refresh-only approach is safer. Alex agreed to add a decision branch in the runbook that lets the engineer choose between refresh-only remediation and full targeted apply based on the dependency graph.

The team reviewed the existing runbook inventory:
- Drift remediation (refresh state + targeted apply) - exists
- State lock force-unlock - exists
- Failed run retry with provider cache clear - exists
- Workspace variable rollback - exists
- Agent pool rebalance - draft, needs testing
- Cross-workspace cascade recovery - does not exist, needs to be built

### RBAC for Remediation Approval

Derek outlined three approval tiers:

| Tier | Actions | Who Can Approve |
|---|---|---|
| Observe | View runs, drift status, AI suggestions | All engineers |
| Approve-Standard | Execute standard runbooks (refresh, re-plan, re-apply, variable rollback) | Senior engineers, SRE leads |
| Approve-Critical | Execute critical runbooks (force-unlock, state rm, targeted destroy, cascade recovery) | SRE leads, platform team lead, infrastructure architect |

James asked how this maps to existing HCP Terraform team permissions. Derek said the remediation agent's RBAC will be a separate layer - it checks the user's role in the agent interface before issuing Terraform commands, which then check the workspace-level team permissions for the actual authorization. Two layers of access control.

Sarah noted that during off-hours, there may be only one senior engineer on call who has standard approval rights. If a critical action is needed, they still need to escalate. Derek confirmed that's intentional - critical actions on production infrastructure should have a second pair of eyes. If it's 3 AM and a state lock needs force-unlocking, the on-call engineer calls him or James. That's a 5-minute phone call, not a 20-minute diagnostic session. The agent already did the hard work of identifying the problem.

### Audit Trail Requirements

Alex walked through the audit model:

- Every AI suggestion logged: timestamp, incident ID, signals used, confidence score, suggested action
- Every engineer action logged: timestamp, incident ID, engineer ID, action taken, reason field, result
- Every runbook step logged: timestamp, step ID, command executed, exit code, output, duration
- Audit log is append-only, written to Datadog with the HCP Terraform audit log integration providing the source-of-truth layer
- All events tagged with organization ID, workspace ID, and run ID for correlation
- Retention: 2 years hot in Datadog, 7 years in Datadog archive tier for compliance

Derek asked about logging AI suggestions that engineers ignore. Alex confirmed those are logged too - the model suggested X, engineer took no action or chose a different path. This data feeds the model feedback loop and compliance reporting.
