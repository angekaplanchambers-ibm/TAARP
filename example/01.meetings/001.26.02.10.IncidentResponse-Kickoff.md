# 01.2026-02-10-IncidentResponse-Kickoff

## TL;DR

Platform engineers spend 30-45 minutes resolving Terraform run failures that an AI agent could reduce to under 15 minutes. The team aligned on building a remediation dashboard for run failures and state conflicts, with AI suggesting root causes and remediation steps that engineers must approve. Prevention, capacity planning, and module registry issues are out of scope for v1.

## Context

Sarah's SRE team manages 47 Terraform workspaces across 3 projects (prod-infrastructure, staging, dev) spanning AWS and GCP. Incident triage relies on switching between HCP Terraform UI, AWS/GCP consoles, Datadog, and Slack. Sarah's team reports that 70% of apply failures and state conflicts follow recognizable patterns, but engineers still perform the same diagnostic sequence manually each time. Priya's team has been exploring whether an AI model trained on historical run data could accelerate triage.

## Key Discussion Points

### Current Incident Workflow Pain Points

Sarah walked through a recent P1 - an apply failure on ws-prod-networking that took 38 minutes from first alert to resolution.

The current flow: engineer gets a PagerDuty alert triggered by a failed run notification, opens HCP Terraform UI to check the run logs, switches to AWS console to check the actual state of the affected resources, checks Datadog for any related infrastructure alerts, then starts correlating provider errors and timestamps manually across these tools. Sarah estimated her team spends 15-20 minutes just gathering signals before they can start diagnosing.

Derek noted that during off-hours, the on-call engineer often lacks the context to correlate signals quickly. They end up pinging a senior platform engineer in Slack, adding another 10-15 minutes of wait time.

Sarah: "We have runbooks, but they're Notion pages nobody opens during a 2 AM page. The engineer's muscle memory is the actual runbook."

### AI-Assisted Triage Concept

Priya presented the concept: an AI agent that ingests HCP Terraform audit log events, run logs, drift detection results, Sentinel policy check results, and notification webhooks in near-real-time. When a failure is detected, it presents the engineer with a correlated view - the relevant signals grouped by probable incident, a confidence-scored root cause hypothesis, and suggested remediation steps.

Derek asked about false positives. Priya acknowledged the model will need tuning and proposed a feedback loop where engineers mark suggestions as helpful or not. The model retrains weekly on this feedback.

Maya asked about the engineer's trust in the AI. If the model suggests a wrong root cause early on, engineers will stop looking at it. The team agreed that the UI needs to show the reasoning chain - which signals led to which conclusion - so engineers can evaluate the suggestion rather than just accept or reject it.

### Dashboard Requirements

Maya walked through initial UX requirements gathered from engineer interviews:

- Single-pane view of failed runs with severity, affected workspaces, and time-in-state
- AI-generated incident summary that updates as new signals arrive
- Signal timeline showing correlated events across HCP Terraform runs, drift detection, and provider errors
- Suggested root cause with confidence score and reasoning chain
- Remediation steps as an actionable checklist, not a paragraph of text
- One-click approval to execute a remediation step, with a required reason field

Derek emphasized that the dashboard must not replace the engineer's judgment. The AI is an accelerator, not an autopilot. Every remediation action requires explicit engineer approval.

Sarah requested that the dashboard show what the AI considered but ruled out - "Sometimes knowing what it's not is faster than proving what it is."

## Decisions & Action Items

| # | Decision | Rationale | Owner |
|---|----------|-----------|-------|
| D1 | Focus on detection-to-resolution flow, not prevention | Prevention is a separate initiative with different data requirements. Mixing them dilutes the v1 scope. | Priya |
| D2 | AI suggests but engineer approves all remediations | Engineers are accountable for infrastructure changes. Autonomous remediation is a trust and compliance risk for v1. | Derek |
| D3 | Start with run failures and state conflicts only | These account for 65% of P1/P2 incidents per Sarah's data. Module registry, provider plugin, and workspace variable issues are v2 candidates. | Sarah |
| D4 | Target 50% MTTR reduction as success metric | Current average MTTR for run failures and state conflicts is 38 minutes. Target is under 19 minutes for AI-assisted triage. | Priya |

## Open Questions

1. What latency is acceptable between signal ingestion and AI suggestion? Sarah mentioned that a 30-second delay would be fine but 5 minutes would make it useless during a cascading failure across workspaces.
2. How do we handle incidents that span multiple workspaces (e.g., a state lock conflict on ws-prod-networking that causes dependent applies in ws-prod-compute to fail)? Single incident or linked incidents?
3. Should the dashboard integrate with the existing PagerDuty workflow or replace the alerting path entirely?
4. What RBAC model applies to remediation approval? Can any engineer approve, or only workspace-level admins per HCP Terraform Teams?
5. How do we handle the AI suggesting a remediation the engineer has never performed? Training gap or guardrail?
